Definition

What is an FPGA?

A Field-Programmable Gate Array is an integrated circuit containing an array of configurable logic blocks (CLBs) connected through programmable interconnects. Unlike a fixed ASIC or a general-purpose CPU, an FPGA can be reprogrammed after manufacture — in the field — to implement any digital circuit the silicon resources allow.

The core insight: instead of hardwiring logic gates, you configure a fabric of lookup tables (LUTs), flip-flops, DSP slices, and on-chip memory, then route signals between them using a software toolchain. The configuration is stored in SRAM and can be reloaded in milliseconds, turning a single chip into a bespoke hardware accelerator for any workload.

"An FPGA is a chip full of open gates that engineers can reprogram as much as needed to add new functionality, adapt to changing protocols, or accelerate algorithms in hardware." — National Inventors Hall of Fame (on Ross Freeman's invention)
History

Origin, inventor, and evolution

The FPGA was invented by Ross Freeman, who co-founded Xilinx in 1984 and filed U.S. Patent No. 4,870,302 in 1985. Freeman was born July 26, 1948 and died October 22, 1989 — just as his invention was beginning to reshape the electronics industry. He was inducted into the National Inventors Hall of Fame in 2009.

Positioning

FPGA as custom ASIC — and how it differs from a GPU

An FPGA occupies a unique position in the compute hierarchy. For any given workload, it behaves like a custom ASIC: optimized datapath, custom bit-widths, deterministic timing. But unlike an ASIC, the "design" is a software bitstream that can be rewritten in hours. And unlike a GPU, the hardware topology itself changes per workload — not just the software running on fixed cores.

FPGA — Reconfigurable custom silicon

Behaves like a custom ASIC for each workload. Reload a bitstream → become a different accelerator. Zero NRE cost, full hardware control, sub-microsecond deterministic latency. The datapath is tuned to the algorithm.

ASIC — Fixed-function chip

Maximum efficiency for one workload in high volume. Optimal power, smallest die — but $10M–$100M NRE, 18-month tape-out cycles, zero post-manufacture flexibility. Model changes require a new chip.

GPU — Programmable throughput engine

Thousands of fixed SIMD CUDA cores running software kernels. Excellent for dense matrix training. High idle power, fixed memory hierarchy, requires large batches for efficiency. Logic topology never changes.

Deep Dive

Inside the
FPGA Fabric

FPGAs are hierarchical arrays of configurable primitives — from individual lookup tables up to hard processor cores and AI tensor engines — all connected by a sea of programmable routing fabric.

Core primitives

The four architectural pillars

Configurable Logic Block (CLB)

The fundamental compute unit. Each CLB contains 4–8 Look-Up Tables (LUTs) implementing arbitrary Boolean functions, flip-flops for state storage, carry chains for fast arithmetic, and multiplexer trees. Modern FPGAs contain millions of CLBs.

Programmable Interconnect

A hierarchical grid of metal wires and switch matrices spanning the die. Short wires connect adjacent tiles; long lines cross half or the full die. The toolchain's place-and-route algorithm determines every connection at compile time — no runtime overhead.

Block RAM (BRAM)

Dual-port SRAM macros embedded throughout the fabric in columns — typically 18 Kb or 36 Kb each. Used for FIFOs, coefficient tables, line buffers, and custom on-chip memory hierarchies. Eliminates off-chip memory latency for working datasets.

DSP Slices

Hard-wired multiply-accumulate (MAC) units running at 500–900 MHz. The AMD VU47P in AWS F2 contains 9,024 DSP slices delivering up to 28 TOPS of INT8 compute performance. The backbone of FPGA neural network acceleration.

Structural diagram

FPGA tile array — top-level view

Each column in the FPGA die is specialized: CLB columns for logic, BRAM columns for memory, DSP columns for arithmetic. The programmable routing fabric wraps every column, enabling any output to connect to any input across the die. I/O and hard-IP blocks ring the periphery.

Modern FPGA enhancements

Hard blocks, AI engines, and HBM

Modern high-end FPGAs embed far more than soft logic. Hard IP blocks provide ASIC-efficiency for common functions without consuming soft-logic resources, while AI Tensor Blocks bring fixed-function matrix accelerators onto the FPGA die itself — blending reconfigurability with hard-IP efficiency.

AI Tensor Blocks

Hard INT8/INT4 multiply-accumulate arrays embedded alongside the LUT fabric. AMD Versal AI and Altera Agilex 9 embed dedicated neural-network engines achieving hundreds of TOPS without consuming soft DSP slices — FPGA + ASIC in one chip.

High Bandwidth Memory (HBM)

Stacked DRAM physically bonded to the FPGA package. AMD VU47P: 16 GiB at 460 GB/s per chip. Eliminates memory bandwidth as the inference bottleneck for large activation tensors and KV-caches.

Hard ARM processor cores

Xilinx Zynq UltraScale+ embeds quad-core Cortex-A53 and dual Cortex-R5 CPUs in the same package. Enables fully standalone FPGA inference systems with hardware Linux running the control plane.

Partial reconfiguration

Reload a section of the FPGA fabric while the rest continues operating. Deploy multiple inference models simultaneously on one chip, switching between them at microsecond granularity without system downtime.

Software-Defined Hardware

Elastic Silicon:
The SDH Paradigm

Software-Defined Hardware treats the FPGA not as a static accelerator, but as a continuously reconfigurable compute substrate co-designed with its ML algorithms and system software as one unified architecture.

Core concept

What is Software-Defined Hardware?

SDH is the philosophy of co-designing ML algorithms, system software stack, and reconfigurable hardware as a single unified system. Instead of mapping a neural network onto fixed GPU cores, SDH allows the hardware datapath itself — bit widths, pipeline depth, memory topology, operator fusion — to be tuned for each model and updated as models evolve.

Concretely: a PyTorch model is compiled not just to a software binary, but to a bitstream that restructures the FPGA's logic fabric for that exact model's compute graph. Change the model → recompile the bitstream → reload the chip in milliseconds. The hardware IS the software.

Key principle

SDH collapses the traditional hardware/software boundary. The FPGA becomes an execution substrate as mutable as software but as efficient as custom silicon. Co-designing ML, OS, and hardware as one system — not three separate stacks — is what delivers 10–100× efficiency gains.

Strategic summary

FPGAs as SDH dominate inference scenarios requiring deterministic latency, custom quantization, high power efficiency, or low batch sizes. GPUs dominate training and high-throughput batch inference. The highest-performing AI infrastructure deploys both in their respective sweet spots.

AI Inference

FPGA as an
Elastic AI Inference Engine

FPGAs deliver cost-efficient, deterministic AI inference by co-designing ML, system software, and reconfigurable hardware as a single unified architecture — eliminating the overhead, idle power, and fixed-topology constraints of GPU-centric inference.

The inference advantage

Why FPGAs excel at AI inference

GPU inference efficiency depends on batch size: large batches amortize fixed-core overhead, but latency grows. Real-world production systems — trading, fraud detection, medical imaging, real-time recommendations — frequently operate at batch=1 or very small batches. FPGAs are natively optimal here: their pipelined architectures process tokens as streaming data with no batch accumulation, no kernel launch latency, no idle power drain.

Custom quantization

Implement 3-bit, 4-bit, or mixed-precision arithmetic natively. A 4-bit LLM requires 4-bit multipliers — FPGA implements exactly that. GPU wastes bits by packing into INT8 containers, burning 2× more compute and memory bandwidth than necessary.

Operator fusion in silicon

Fuse LayerNorm + Attention + Softmax + FFN into a single hardware pipeline. No off-chip memory round-trips between operators. Latency collapses from tens of microseconds on GPU to nanoseconds on FPGA.

Deterministic SLA

Hardware pipelines have fixed, clock-cycle-accurate latency. No scheduler jitter, no GC pauses, no CUDA launch overhead. Critical for financial trading, autonomous vehicles, and medical devices with hard latency budgets.

Elastic capacity

AWS F2 instances scale from 1 FPGA (f2.6xlarge) to 8 FPGAs (f2.48xlarge). Hot-swap a new bitstream to switch models in milliseconds. Pay for exactly the compute you need with per-second billing.

Decision guide

When FPGAs win at inference

Latency-critical real-time applications

When SLA demands sub-millisecond response: HFT, fraud scoring at transaction time, autonomous vehicle perception, real-time sensor fusion. GPUs cannot match FPGA pipeline latency at batch=1.

Power-constrained environments

Edge deployments, satellite computing, dense inference racks where per-inference watt budget is critical. FPGA: 10–25W total. H100 GPU: 700W. 28–70× power difference at comparable throughput for specific workloads.

Non-standard model precision

Ternary networks, 4-bit LLMs, 2-bit activations. GPU packs sub-8-bit values into INT8 containers, wasting half the compute. FPGA implements the exact arithmetic natively — 2× to 4× efficiency gain.

Streaming / continuous data

Video streams at 60fps, 100 Gbps packet flows, continuous genomics pipeline. FPGA processes each sample as it arrives — no batch accumulation, no head-of-line blocking, no latency variance.

Frequent model updates in production

A/B testing hardware-level inference logic, regulatory-driven model changes, rapid iteration cycles. FPGA bitstream hot-swap requires no system restart — update the accelerator logic in milliseconds.

When to prefer GPU

High-throughput batch inference (>128 batch size), models larger than ~80 GiB, large-scale training, teams without FPGA expertise, and rapid prototyping without FPGA toolchain setup. GPU's CUDA ecosystem is unmatched for development velocity.

Advanced Architecture Patterns

Next-Generation
Disaggregated AI Inference

Beyond single-chip deployments, the frontier of FPGA AI inference uses disaggregated architectures — splitting different inference phases across specialized hardware to extract maximum efficiency from each component.

Disaggregated inference

FPGA for Prefill + Trainium for Decode

Large language model inference has two phases with fundamentally different compute profiles. Prefill processes the input prompt — highly parallel, compute-bound, with variable-length inputs. Decode generates tokens autoregressively — memory-bandwidth-bound, sequential, highly regular. Disaggregating these phases onto purpose-built hardware maximizes efficiency for each.

Phase 1 — Compute-bound
FPGA Prefill

Custom attention pipelines on VU47P HBM fabric. Processes full prompt in parallel with bespoke datapath optimized for variable-length inputs. HBM at 460 GB/s eliminates KV-cache bandwidth bottleneck. Deterministic latency regardless of prompt length variance.

→
KV Cache transfer
High-Speed Fabric

RDMA / PCIe Gen4 or AWS EFA moves KV-cache tensors from FPGA prefill nodes to decode accelerator. AWS F2 inter-FPGA ring: 400 Gbps bidirectional bandwidth at 12 Gbps per FPGA pair.

→
Phase 2 — Bandwidth-bound
Trainium / Inferentia Decode

AWS Trainium NeuronCore architecture is purpose-built for memory-bandwidth-bound autoregressive generation. Large HBM pool holds full model weights; decode proceeds at maximum memory bandwidth efficiency.

Why disaggregation wins

Prefill is arithmetic-intense with irregular parallelism — FPGA custom attention handles variable-length prompts without batch-padding waste. Decode is memory-bandwidth-limited with regular patterns — Trainium's NeuronCore is purpose-built for this. Combining both achieves lower total cost of ownership than running H100s for the entire pipeline, with no compromise on latency or throughput.

Advanced patterns

Emerging FPGA AI architectures

Inline data-path acceleration

FPGA placed directly in the data stream — between network ingress and CPU/GPU. Pre-processes, filters, and infers at line rate (100 Gbps) before data touches a general-purpose processor. Eliminates memory bandwidth pressure on expensive downstream compute.

Speculative decoding offload

Small FPGA-accelerated draft model speculatively generates token candidates (microseconds/token). GPU verifier model accepts valid sequences in batches. Result: 3–5× decode throughput improvement for LLMs without accuracy loss.

Mixture-of-Experts routing

FPGA implements the MoE expert router in hardware: each token triggers a sub-microsecond lookup table dispatch to the correct expert shard on the GPU cluster. Eliminates routing overhead that bottlenecks CPU-based MoE systems.

Chiplet-integrated FPGA

FPGA reconfigurable fabric as a chiplet co-packaged with a CPU die via UCIe or EMIB. Sub-nanosecond die-to-die latency, cache coherence with CPU. Merges FPGA flexibility with CPU memory hierarchy — the next frontier of adaptive compute.

Smart NIC / DPU-FPGA hybrid

FPGA fabric embedded in the network interface card handles encryption, compression, and ML inference at line rate before data reaches the server. Effectively moves AI to the network edge — zero additional data-center floor space.

Open-Minded AI Infrastructure

AI Inference is
Not GPU-Only

The dominant narrative equates AI inference with GPU inference. This is historically contingent — shaped by CUDA's developer momentum and the training era's needs — not technically inevitable. The future of AI infrastructure is heterogeneous.

The case for heterogeneity

Why the GPU monoculture has limits

GPUs won the training era because deep learning maps well onto SIMD parallelism, and NVIDIA's CUDA ecosystem created deep lock-in. But production inference is a different problem: latency matters more than peak throughput, batch sizes are often small, power budgets are tight, models update frequently, and many workloads require custom precision formats that don't fit cleanly into INT8 containers.

An infrastructure optimized purely for training throughput is not automatically optimal for inference efficiency. The first-principles question is always: what hardware topology — pipeline depth, memory hierarchy, compute precision, datapath width — best matches this workload's compute graph?

"The history of compute is the history of specialization. CPUs begat GPUs begat TPUs begat FPGAs in the data center. Each wave did not replace the last — it found the workload it was built for."

Key takeaways

What we've learned across this guide

FPGAs are reconfigurable ASICs

For any given workload, an FPGA can match ASIC-level efficiency without the NRE cost or inflexibility. The bitstream IS the chip design — rewritable in hours, deployable in milliseconds.

Software-Defined Hardware is real and powerful

Co-designing ML, system software, and hardware as one system — rather than adapting models to fixed GPU cores — yields 10–100× efficiency improvements for specific inference workloads.

AWS democratizes FPGA at cloud scale

F1 and F2 instances let any team prototype, validate, and deploy FPGA accelerators without capital expenditure. F2 with HBM delivers 60% better price/performance than F1 — and AWS handles all operational underpinning.

Disaggregated inference is the future

FPGA prefill + Trainium decode + GPU training is not a workaround — it is an optimally designed heterogeneous system. Each phase maps to the silicon it was built for. Total cost of ownership drops; performance rises.

The bottom line

Open-minded AI infrastructure teams evaluate each workload phase on its own merits — latency requirements, batch size distribution, precision needs, power budget, update frequency — and match each phase to its optimal accelerator. GPU, FPGA, and custom ASIC are complementary tools, not competing choices. The question is never "GPU or FPGA?" — it is "what hardware topology does this compute graph require?"
