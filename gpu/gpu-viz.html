Chapter 01

The Machine

A modern GPU is a massively parallel processor built around thousands of small cores organized into Streaming Multiprocessors. Understanding its physical layout is the foundation of everything that follows.

What is a GPU?

A GPU (Graphics Processing Unit) is optimized for throughput, not latency. While a CPU has a handful of powerful cores that minimize time per operation, a GPU trades latency for massive parallelism — thousands of simpler cores doing the same operation on different data simultaneously.

CPU vs GPU philosophy

A CPU is a Formula 1 car — blazing fast on a single lap. A GPU is a fleet of buses — slower individually, but moving thousands of passengers at once. Deep learning is a traffic problem, not a race.

The Streaming Multiprocessor (SM)

Every SM is a self-contained processor. It has its own CUDA cores, warp schedulers, register file, and shared memory. Modern SMs contain 64–128 CUDA cores, 4 warp schedulers, 256KB register file, and 64–228KB shared memory / L1 cache per SM.

SM INTERNAL LAYOUT

Warp schedulers — why 4?

Each SM has 4 warp schedulers that can issue instructions every clock cycle. With typical memory latency of 400–800 cycles, the scheduler fills the pipeline by switching to other warps while one waits for memory — a technique called latency hiding. More warps in flight = better latency hiding.

The golden rule of GPU programming

Move data as close to the compute as possible. Every time you spill from registers to shared memory, or from shared memory to L2, you pay a latency and bandwidth tax. The biggest performance wins in CUDA kernels almost always come from reducing trips to global memory — not from making the math faster.

SIMT — Single Instruction, Multiple Threads

All 32 threads in a warp fetch the same instruction simultaneously. Each has its own register state and can branch independently — but when branches differ, the hardware serializes them. This is warp divergence (covered in detail in Chapter 4).

WARP SCHEDULER — LATENCY HIDING IN ACTION

When Warp A stalls on memory (yellow), the scheduler instantly switches to Warp B. This is essentially free — each warp's registers stay on the SM the entire time.

Why 32?

32 threads per warp is an NVIDIA architectural choice dating to the first CUDA GPUs. It balances the cost of the warp state (registers) against the benefit of SIMT execution. AMD uses "wavefronts" of 64 threads.

Warp state is always resident

Unlike CPU context switches, warp switching is zero-cost — the SM keeps ALL active warp registers in its register file simultaneously. Switching is just pointing the scheduler at a different warp's already-loaded registers.

Occupancy — filling the SM pipeline

Occupancy = (active warps per SM) / (max warps per SM). Higher occupancy means more warps available to hide latency. But maximum occupancy isn't always optimal — sometimes fewer, better-utilizing warps beat many idle ones.

The occupancy vs IPC tradeoff

Using fewer registers per thread lets more warps co-reside (higher occupancy). But using too few registers forces the compiler to spill to local memory — which is actually global memory in disguise, tanking performance. Profile with nsight-compute, not intuition.

Chapter 03

Memory Coalescing

When 32 threads in a warp access memory, the hardware tries to merge those 32 individual requests into as few 128-byte cache line transactions as possible. The difference between coalesced and scattered access can be 10–32× in performance.

COALESCING SIMULATOR — 32 THREADS ACCESSING MEMORY
Global memory — 128 cells shown (each = 4 bytes)
Thread access pattern (32 threads → which address?)

The 128-byte cache line rule

Global memory is served in 128-byte chunks. If all 32 warp threads access addresses within the same 128-byte block, that's 1 transaction. If each thread accesses a completely different cache line, that's up to 32 separate transactions — a 32× bandwidth penalty.

Real-world impact

A matrix transpose naively written causes every warp to read a column (non-coalesced). Rewriting with shared memory tiling as a staging buffer converts it to row-major reads and writes — turning 25 GB/s effective bandwidth into 750+ GB/s on an A100.

Common access patterns and their cost

✓ Coalesced — row-major matrix read

Threads 0–31 read A[row][0..31]. All within one 128-byte line. 1 transaction for the entire warp.

// GOOD: thread i reads element i (consecutive) output[tid] = data[tid]; // 1 transaction
1 transaction100% BW efficiency

⚠ Strided — partial coalescing

Threads read every 2nd element. Each pair of threads shares a cache line — 16 transactions instead of 1.

// WARN: stride 2 — wastes half the cache line output[tid] = data[tid * 2]; // ~16 transactions
16 transactions50% BW efficiency

✗ Column-major — worst case

Classic transpose bug: thread i reads A[i][j], traversing rows. Every thread touches a different 128-byte chunk.

// BAD: each thread hits a separate cache line output[tid] = data[tid * N]; // up to 32 transactions
32 transactions~3% BW efficiency

The tiling pattern — shared memory as a staging buffer

Load data from global memory in coalesced chunks into shared memory (fast), then read from shared memory in any pattern you need. This is the foundation of optimized matrix multiplication (GEMM), convolutions, and almost every high-performance CUDA kernel.

__shared__ float tile[32][32]; // on-chip, 4KB // Step 1: coalesced load from global → shared tile[threadIdx.y][threadIdx.x] = A[row * N + col]; __syncthreads(); // Step 2: read shared in any order — no global memory cost float val = tile[threadIdx.x][threadIdx.y]; // transposed! // Step 3: coalesced write back to global B[col * M + row] = val;
MATRIX TRANSPOSE WITH TILING

Bank conflicts in shared memory

Shared memory is divided into 32 banks (one per warp lane). If two threads in a warp access the same bank (but different addresses), the accesses serialize — a "bank conflict." Classic fix for matrix transpose: pad the shared array by +1 column to offset addresses: float tile[32][33] instead of [32][32].

Chapter 04

Warp Divergence

Because all 32 threads in a warp execute the same instruction in lockstep, when they need to take different branches, the hardware must serialize those paths — executing each branch with non-participating threads masked off.

The hidden cost

Masked threads aren't free — the warp slot stays occupied for the duration of each serialized path. Inactive lanes consume warp scheduler time, register file space, and issue bandwidth. They just don't produce results.

Worst case: checkerboard

If odd threads take branch A and even take B, every other lane is masked for every instruction. A 32-thread warp effectively runs as two back-to-back 16-thread half-warps — 2× the cycles for the same work.

Strategies to minimize divergence

1. Branch on block-level invariants

If all 32 threads in a warp always take the same branch, there's no divergence — even if different warps take different branches. Structure conditionals so the predicate is identical within a warp.

// Good: warp-level invariant, blockIdx decides if (blockIdx.x % 2 == 0) { doA(); } else { doB(); }

2. Predicated execution

For short branches, the compiler uses predication — both paths compile to the same instructions with conditional masks. If both paths are cheap, predication avoids the overhead of serialization entirely.

// Compiler may predicate this if bodies are small val = (threadIdx.x < 16) ? A[tid] : B[tid];

3. Data reorganization (sorting)

Sort input data so threads with similar execution paths end up in the same warp. Used in ray tracing (sort rays by material), sparse operations (sort by sparsity pattern), and BVH traversal.

4. Warp-level primitives (shuffle)

Use __ballot_sync and __shfl_sync to pass data between lanes within a warp without shared memory — reducing branch count by sharing work across lanes.

Chapter 05

Synchronization

GPU threads within a block can cooperate through shared memory — but only if they synchronize at barriers. Understanding the levels of synchronization and their costs is essential for correct, high-performance kernels.

__syncthreads() — the block barrier

Every thread in the block must reach __syncthreads() before any can proceed. This ensures shared memory written by one thread is visible to all others. Calling it inside a divergent branch is undefined behavior — all threads must call it or none.

BARRIER VISUALIZATION — 8 THREADS, ONE BLOCK
__syncthreads() barrier

__syncwarp() — warp barrier

Faster than __syncthreads() — synchronizes only the 32 threads in a warp. Required since Volta (independent thread scheduling) when you need warp-level memory consistency after divergent writes.

cooperative_groups::sync()

CUDA 9+ API for flexible synchronization across arbitrary thread groups — subsets of a block, the entire grid (with caveats), or even multi-GPU. Enables recursive algorithms and producer-consumer patterns.

__shared__ float smem[256]; // Phase 1: all threads load into shared memory smem[threadIdx.x] = globalData[blockIdx.x * 256 + threadIdx.x]; __syncthreads(); // ← BARRIER: wait for ALL threads to finish loading // Phase 2: now safe to read smem written by OTHER threads float neighbor = smem[(threadIdx.x + 1) % 256]; result[threadIdx.x] = smem[threadIdx.x] + neighbor;

Atomic operations — when threads write the same address

Atomics guarantee read-modify-write operations are indivisible — no two threads can interfere. Used for histograms, counters, and reduction. But atomics to the same address serialize — a counter updated by 10,000 threads becomes a 10,000-step pipeline.

ATOMIC VS RACE CONDITION
Without atomics — data race
// Thread A: reads counter=5 // Thread B: reads counter=5 // Thread A: writes 5+1=6 // Thread B: writes 5+1=6 ← LOST! counter++; // count=6, not 7
Undefined behavior
With atomics — safe
// Hardware serializes the RMW: // Thread A atomically: 5→6 // Thread B atomically: 6→7 atomicAdd(&counter, 1); // count=7 ✓
Correct result

Reducing atomic contention

Instead of all threads hitting one global counter, use a two-phase approach: each thread accumulates locally into shared memory using fast atomics (no off-chip traffic), then one thread per block commits a single atomic to global memory. This turns N global atomics into N/blockSize global atomics.

__shared__ int localCount[1]; if (threadIdx.x == 0) localCount[0] = 0; __syncthreads(); // Each thread increments local counter (fast — shared mem atomics) atomicAdd(&localCount[0], myContribution); __syncthreads(); // Only one thread per block hits global memory if (threadIdx.x == 0) atomicAdd(globalCounter, localCount[0]); // 1 global atomic per block

Parallel reduction — the canonical sync pattern

Sum N numbers using ⌈log₂ N⌉ steps instead of N steps. Each step halves the number of active threads with a barrier in between. This is the foundation of every reduction in deep learning: LayerNorm, Softmax, loss computation.

TREE REDUCTION (8 values)

Press Step to simulate each synchronization round.

// Parallel reduction in shared memory for (int stride = blockDim.x / 2; stride > 0; stride >>= 1) { if (threadIdx.x < stride) { smem[threadIdx.x] += smem[threadIdx.x + stride]; } __syncthreads(); // barrier between EVERY step } // smem[0] now contains the sum

Warp-level reduction (no sync needed)

For the final 32 elements (one warp), you can use __shfl_down_sync instead of shared memory — passes data between lanes at register speed, no synchronization needed within a warp.

Grid-level sync (CUDA 9+)

cooperative_groups::grid_group allows synchronizing all blocks in a grid — enabling algorithms that previously required kernel re-launches. Used in single-pass LayerNorm and flash attention variants.
