Overview

What's New in Trainium2

Trainium2 is a ground-up redesign of the Trainium chip — 4× faster, 4× more memory bandwidth, and 3× more HBM capacity than its predecessor. It introduces the third-generation NeuronCore-v3, HBM3 memory, Logical NeuronCore Configuration, NeuronLink-v3 intra-node interconnect, and the entirely new Trn2 UltraServer form factor.

⚡

4× Compute Leap

Each Trainium2 chip delivers 1.3 dense FP8 PFLOPS — a 4× increase over Trainium1. The trn2.48xlarge reaches 20.8 PFLOPS with 16 chips, while UltraServer scales to 83.2 PFLOPS across 64 chips.

1.3 PF/chip (dense FP8)
🧠

8 NeuronCore-v3 per Chip

Trainium2 quadruples core density to 8 NeuronCore-v3 engines per chip (up from 2 in Trn1). Cores can be addressed individually or merged into Logical NeuronCores for larger effective memory pools.

8 NCs per chipLNC support
💾

96 GB HBM3 per Chip

Triple the memory capacity of Trainium1 with 96 GB of HBM3 per chip running at 2.9 TB/s — a 4× increase in bandwidth. Each trn2.48xlarge instance has 1.5 TB of shared HBM at 46 TB/s aggregate.

96 GB HBM32.9 TB/s per chip

NeuronLink-v3 Interconnect

New NeuronLink-v3 provides 1 TB/s chip-to-chip bandwidth in a 4×4 2D Torus intra-instance topology. UltraServer extends this with ring connections between the same-coordinate chips across four instances.

Elastic Fabric Adapter v3 delivers 3.2 Tbps per instance — doubling EFAv2 — with 35% lower collective-communication latency. UltraServers scale to 12.8 Tbps. Encryption in transit at no performance penalty via Nitro.

HBM queue reordering: Trainium2 introduces automated reordering of the HBM memory queue in hardware, achieving up to 95% HBM bandwidth utilization — a dramatic improvement over typical 60–70% utilization in systems that rely on sequential request ordering. This directly benefits memory-bandwidth-bound operations like large attention mechanisms in LLM training.

Compute Engine

NeuronCore-v3 Architecture

NeuronCore-v3 is the third-generation AWS ML compute engine. It retains the proven four-engine design from NeuronCore-v2 — Tensor Engine, Vector Engine, Scalar Engine, and GpSimd Engine — while adding significant capability improvements including larger SBUF, hardware-based DMA descriptor generation, and improved memory bandwidth utilization.

NeuronCore-v3 — Compute Engines & Memory

🔢 Tensor Engine (TensorE)

Systolic array for matrix multiplications (GEMM), convolution, and transposition. Reads inputs from SBUF, outputs partial sums to PSUM. Loops over the K-dimension to complete full matrix multiplications. Primary engine for large transformer attention and feed-forward layers.

📊 Vector Engine (VectorE)

SIMD-parallel element-wise operations with multiple inputs to multiple outputs. Handles activation functions, layer normalization, dropout, and other parallelizable element-wise workloads. Higher throughput per cycle for broadcast-style operations.

⚡ Scalar Engine (ScalarE)

Hardware-accelerated 1:1 element scalar ops. Provides native acceleration for transcendentals like GeLU, Sqrt, Exp, and Log — eliminating software emulation on the CPU or GpSimd. Critical for efficient activation function throughput.

🛠 GpSimd Engine

Eight Turing-complete SIMD processors, each with an integrated DMA engine that can access any SBUF or HBM on-chip or across all chips in the same Trn2 instance. Total integrated DMA bandwidth: 307 GB/s (153 GB/s per direction). Enables custom C++ operators without CPU round-trips.

Hardware-Based DMA Descriptor Generation

Trainium2 introduces a new hardware DMA Descriptor Generator Engine (DGE). Prior generations relied on static DGE (pre-generated at load time, stored in HBM) or software DGE (generated dynamically on GpSimd, consuming SBUF space). The new hardware DGE generates descriptors on-demand without additional memory storage, and frees GpSimd for useful computation.

NEW in Trn2No HBM overhead

HBM→SBUF DMA Transpose

NeuronCore-v3 adds hardware-accelerated DMA transpose operations: data can be transposed in-flight during transfer from HBM to SBUF. This is essential for LLM decode where the K-cache has layout [seqlen, d_head] in HBM but TensorE requires d_head as the partition dimension for Q@K attention.

Logical NeuronCore Configuration (LNC)

Beginning with Trainium2, AWS Neuron introduces Logical NeuronCore Configuration (LNC). LNC lets you combine the compute and memory resources of multiple adjacent physical NeuronCores into a single larger logical NeuronCore — directly improving efficiency for models where larger working memory per "device" reduces communication overhead.

When to Use LNC2

Use Logical NeuronCore Configuration (LNC2) when your model's working set per shard is large enough that a single physical NeuronCore's 12 GB HBM becomes a bottleneck. LNC2 doubles the effective HBM to 24 GB and SBUF to 56 MiB per "device," reducing tensor parallelism degree required and the associated communication overhead.

Large modelsReduce TP degree

Performance Trade-off

LNC2 reduces the number of addressable logical NeuronCores from 128 to 64 per trn2.48xlarge. Compute resources are pooled but not doubled — the benefit is larger per-core memory at the cost of reduced aggregate parallelism granularity. The compiler handles LNC assignment automatically when configured.

64 LNCs per instanceCompiler-managed

LNC and Tensor Parallelism: With LNC2 enabled, each logical NeuronCore holds 24 GB of HBM — sufficient for a complete Llama-70B layer in BF16 without sharding across NeuronCores. This allows a Tensor Parallelism degree of 4 per instance instead of 8, halving the AllReduce communication overhead for each forward pass. For inference on Trn2, this translates directly to lower per-token latency.

Dual Topology Design

Within each instance, 16 Trainium2 chips are arranged in a 4×4 2D Torus topology via NeuronLink-v3. Each chip connects to its horizontal and vertical neighbors with wrap-around edges — enabling efficient AllReduce for tensor parallelism.

Across instances, chips at the same (X,Y) coordinates in each of the four instances are connected in a ring topology via NeuronLink-v3 Active Electrical Cables (AEC/OSFP-XD). This forms a depth-4 ring of 16 "columns" of 4 chips each.

The combined topology enables efficient 3D parallelism: Tensor Parallelism within 2D Torus columns, Pipeline Parallelism along the ring depth, and Data Parallelism across UltraServers.

Numerical Precision

Supported Data Types

Trainium2 supports the same rich set of numerical formats as Trainium1, now accelerated by NeuronCore-v3's higher-throughput compute engines. The combination of dense and sparse FP8 modes gives Trainium2 exceptional flexibility for both training accuracy and inference speed.

cFP8

Configurable FP8 — AWS-native format with tunable exponent/mantissa split per-layer. 1,300 TFLOPS dense per chip. Used for production model training where accuracy must be preserved.

Dense: 1.3 PFLOPS

Sparse FP8

Sparse computation mode for FP8, reaching 5,200 TFLOPS per chip — 4× dense throughput. Effective for inference and fine-tuning phases where activations have natural sparsity.

Sparse: 5.2 PFLOPS

BF16

667 TFLOPS BF16 per chip. The preferred format for large-scale LLM training with optimal numerical stability for gradient accumulation and Adam optimizer states.

667 TFLOPS/chip

FP16

IEEE 754 half-precision with hardware stochastic rounding support. Compatible with GPU-trained checkpoints and frameworks. Used in mixed-precision training with FP32 master weights.

Stochastic rounding

TF32

TensorFloat-32 with 10-bit mantissa and 8-bit exponent. Near-FP32 accuracy at significantly higher throughput. Used in mixed-precision training where full FP32 accuracy is not needed.

TF32 compute

FP32

Full single-precision for master weight copies, gradient buffers, and optimizer states. Used with mixed-precision training where the model forward pass runs in BF16/FP16.

Master weights

INT8 / UINT8

8-bit integer quantization formats for inference. Trainium2's NeuronCore-v3 delivers significantly higher INT8 throughput than NCV2, enabling cost-efficient quantized inference serving.

Inference quantization

Hardware Stochastic Rounding

All low-precision formats benefit from hardware-accelerated stochastic rounding, preventing systematic bias accumulation in gradient updates. No additional software overhead or SBUF usage.

Hardware SR · all types

The AWS Neuron SDK on Trainium2 adds native support for LNC configuration, hardware DMA transpose, GpSimd integrated DMA, and the UltraServer multi-instance topology — all accessible through PyTorch NeuronX, JAX, and the NxD Training/Inference libraries.

NKI: Neuron Kernel Interface

Trainium2 is the primary target for NKI — a Python-based kernel programming API similar to Triton for CUDA. NKI lets engineers write custom compute kernels that execute on NeuronCore-v3 hardware engines, including the GpSimd integrated DMA for SBUF/HBM access patterns not covered by the compiler.

# NKI kernel using dma_transpose for K-cache import neuronxcc.nki.language as nl import neuronxcc.nki.isa as nisa @nki.jit def attn_kernel(q, k, v, out): # Transpose K from [seqlen, d_head] → SBUF layout k_sbuf = nisa.dma_transpose(k) scores = nl.matmul(q, k_sbuf) # Q @ K^T ...
