Overview

Why Trainium3 Is Different

Trainium3 (Trn3) is AWS's most ambitious chip generation yet — the first to use a 3nm process, the first to introduce a dedicated switch silicon (NeuronSwitch-v1), and the first designed for rack-scale all-to-all topology at 144 chips. It delivers 2× compute and 1.7× memory bandwidth over Trainium2 per chip, and 4.4× higher UltraServer-level performance.

⚡

First 3nm AWS AI Chip

Trainium3 is fabricated on TSMC's N3P 3nm-class process — the most advanced node used in any shipping AI accelerator. Dual compute chiplets are co-packaged with four HBM3e stacks via a high-bandwidth proprietary die-to-die interface, achieving 2.52 PFLOPS of dense MXFP8 compute per package.

TSMC N3PDual-chiplet
🔀

NeuronSwitch-v1: All-to-All Fabric

The breakthrough in Trn3 is NeuronSwitch-v1 — a dedicated switch chip that creates a full all-to-all topology across all chips in an UltraServer. This replaces the 2D Torus + ring topology of Trn2 UltraServer, slashing communication latency for Mixture-of-Experts and autoregressive inference where any-chip-to-any-chip bandwidth is critical.

📈

4.4× UltraServer Performance

Compared to Trn2 UltraServers, Trn3 Gen2 UltraServers deliver 4.4× more compute, 3.9× more memory bandwidth, 4× better energy efficiency, and 5× higher output tokens per megawatt on Amazon Bedrock at similar user latency. GPT-OSS 120B showed 3× higher per-chip throughput and 4× faster response times.

MXFP8 & MXFP4 Support

Trainium3 introduces Microscaling (MX) data formats — MXFP8 and MXFP4 — as first-class hardware data types. MXFP8 enables ultra-efficient training with shared block scaling, while MXFP4 enables 2× the compute density for inference tasks where 4-bit quantisation is acceptable.

Compute Engine

NeuronCore-v4 Deep Dive

NeuronCore-v4 retains the proven four-engine design (Tensor, Vector, Scalar, GpSimd) while introducing a dramatically enlarged Tensor Engine with a dual systolic array design — a 128×128 BF16 array and a new 512×128 MXFP8/MXFP4 array — plus 32 MiB of on-chip SRAM, 33% more than NCV3.

NeuronCore-v4 — Four Compute Engines

🔢 Tensor Engine — Dual Systolic Array

Two systolic arrays in one NeuronCore: a 128×128 BF16 array identical in size to Trn2 for high-accuracy training, and a new 512×128 MXFP8/MXFP4 array that is 4× wider for low-precision dense and sparse compute. Both arrays read inputs from SBUF and write partial sums to PSUM. The K-dimension looping enables full matmul completion in a single NeuronCore with no inter-core data movement for weight-stationary workloads.

📊 Vector Engine (VectorE)

SIMD-parallel element-wise operations processing multiple inputs to multiple outputs in a single cycle. Handles activations (GeLU, SiLU, ReLU), layer normalization, embedding lookups, dropout, and other vector-parallelizable operations. Improved throughput on Trn3 for exponential operations — directly accelerating softmax in attention mechanisms.

⚡ Scalar Engine (ScalarE)

Hardware-accelerated 1:1 element operations. Provides native execution of transcendental functions including Exp, Log, Sqrt, and reciprocal — eliminating the need for GpSimd emulation and freeing those cycles for custom operator work. Essential for efficient SiLU and Swish activations used in modern LLM feed-forward layers.

🛠 GpSimd Engine — 128 DMA Engines

Turing-complete programmable SIMD processors exposing the full Trainium ISA via the Neuron Kernel Interface (NKI). Each GpSimd processor has direct access to SBUF and HBM across all chips in an instance, with 128 shared hardware DMA engines providing high-bandwidth data movement for custom kernel patterns not covered by the graph compiler.

Dual Systolic Array Design

The defining architectural change in NeuronCore-v4 is the addition of a second, wider systolic array alongside the standard BF16 array. The 512×128 MXFP8/MXFP4 array has 4× the width of the BF16 array, enabling proportionally higher throughput for lower-precision workloads. Both arrays coexist and can be scheduled by the compiler to overlap BF16 and MXFP8 compute.

128×128 BF16512×128 MXFP8

LNC — Logical NeuronCore Configuration

Trainium3 ships with Day-0 support for LNC=1 (each physical NC used individually) and LNC=2 (two adjacent physical NCs fused into one logical NC with 64 MiB SRAM and 36 GB HBM per logical core). LNC=8 (fusing all 8 NCs into one massive logical core with 256 MiB SRAM and 144 GB effective HBM) is planned for mid-2026 and is the format most favoured by ML researchers for large sequence lengths and very wide layers.

Gen2 Topology Detail

In Gen2, each of the 36 servers hosts 4 Trainium3 chips. Intra-server chips connect via a first-level NeuronSwitch-v1. Cross-server connectivity goes through two second-level NeuronSwitch-v1 switches, creating a 3-tier hierarchy: chip → L1 switch → L2 switch. Despite this hierarchy, the all-to-all logical topology is preserved — any chip can reach any other chip in O(1) switch hops at full bandwidth.

Interconnect Innovation

NeuronSwitch-v1 & NeuronLink-v4

The biggest architectural leap from Trn2 to Trn3 is the introduction of NeuronSwitch-v1 — a dedicated switch silicon that replaces the fixed 2D Torus + ring topology with a fully programmable all-to-all fabric. Combined with NeuronLink-v4 running on PCIe Gen 6, this doubles per-chip interconnect bandwidth and fundamentally changes what collective communication patterns are efficient at scale.

Why All-to-All Matters

Trn2 UltraServer used a 2D Torus (intra-instance) + Ring (inter-instance) topology — efficient for ring-based AllReduce but suboptimal for Mixture-of-Experts (MoE) all-to-all routing where each token's hidden state must be routed to an arbitrary expert on any chip.

NeuronSwitch-v1 creates a full non-blocking all-to-all topology — any chip can send data to any other chip at full NeuronLink-v4 bandwidth without traversing intermediate hops. This makes expert-parallel training and autoregressive inference (where KV-cache shards on arbitrary chips must be fetched) significantly more efficient.

MoE on Trn3 UltraServer: Mixture-of-Experts models like Mixtral 8×22B, DeepSeek V3, and GPT-4-style MoE architectures require all-to-all communication for expert routing in every forward pass. On Trn2's torus topology, this required multi-hop routing adding latency. On Trn3's NeuronSwitch-v1 fabric, every expert-to-expert transfer is a single switch hop at full 2 TB/s bandwidth — dramatically improving MoE training throughput and enabling models with hundreds of experts to scale efficiently across the full 144-chip Gen2 UltraServer.
Numerical Precision

Advanced Data Types

Trainium3 introduces the MXFP8 and MXFP4 Microscaling formats as first-class hardware types — a significant upgrade over the custom cFP8 on prior generations. These standard MX formats enable interoperability with other MX-compliant hardware and future-proof training pipelines.

MXFP8

Microscaling FP8 — IEEE-standard block-scaled 8-bit format with shared exponent per 32-element block. Enables 2.52 PFLOPS dense throughput. Primary training format for frontier models, offering better accuracy than cFP8 with hardware block-scaling engines.

2.52 PF/chip densePrimary training

MXFP4

Microscaling FP4 — 4-bit block-scaled format enabling 2× the compute density of MXFP8 via the 512×128 systolic array's 4-bit path. Targeted at inference and quantisation-aware training where 4-bit weights/activations are acceptable.

5.0 PF/chipInference & QAT

BF16

BFloat16 via the 128×128 systolic array. The highest-accuracy training format on Trn3, ~671 TFLOPS per chip. Used for master weights, gradient accumulation, and precision-critical training runs. Unchanged array size from Trn2.

~671 TFLOPS/chipMaster weights

FP16

IEEE 754 half-precision with hardware stochastic rounding. Compatible with GPU-trained checkpoints for fine-tuning from NVIDIA-trained models. Runs on the BF16 systolic array with appropriate rounding applied at data boundaries.

HW stochastic round

FP32

Full 32-bit precision for optimizer states, loss scaling denominators, and any computation where full numerical range is required. Used in conjunction with BF16/MXFP8 in mixed-precision training loops.

~183 TFLOPS/chipOptimizer states

4× Structured Sparsity

Hardware-accelerated 16:4 structured sparsity — 4 non-zero values per 16-element block — delivering up to 4× compute throughput over dense computation. Enables sparse MoE expert weight pruning without accuracy loss for inference serving.

16:4 pattern4× throughput

HW Stochastic Rounding

All low-precision data types benefit from hardware stochastic rounding — preventing systematic bias accumulation that degrades convergence in long training runs. Implemented in hardware with no memory or performance overhead on Trn3.

All formatsZero overhead

cFP8 (Legacy)

Configurable FP8 from prior generations remains supported for backward compatibility with Trn1/Trn2 model checkpoints and training recipes. New workloads should prefer the standardized MXFP8 format for better interoperability.

Backward compatTrn1/Trn2 ckpts
Developer Platform

AWS Neuron SDK for Trn3

The AWS Neuron SDK on Trainium3 exposes the full NeuronCore-v4 ISA through NKI, supports MXFP8/MXFP4 natively in the compiler, and integrates with the complete ML framework ecosystem — enabling frontier model training and inference without code changes from Trn2.

Neuron Kernel Interface (NKI)

NKI is the primary differentiator on Trn3 for performance engineering. It exposes the full Trainium ISA — including instruction-level scheduling, SBUF/PSUM memory allocation, engine-level compute dispatch, and DMA management — via Python syntax, similar to Triton for CUDA.

# NKI: custom MXFP8 flash-attention kernel on Trn3 import neuronxcc.nki.language as nl import neuronxcc.nki.isa as nisa @nki.jit def flash_attn_mxfp8(q, k, v, output): # Tile over sequence dimension in SBUF (32 MiB) q_tile = nl.load(q[0:128, :]) # → SBUF k_tile = nisa.dma_transpose(k) # HBM→SBUF + transpose # MXFP8 matmul on 512×128 systolic array scores = nisa.nc_matmul( q_tile, k_tile, dtype=nl.mxfp8 ) scores = nl.softmax(scores, axis=-1) ...

Neuron Explorer — Full-Stack Profiling

Neuron Explorer connects PyTorch / JAX code all the way down to hardware engine operations — identifying sharding inefficiencies, kernel bottlenecks, and NeuronLink utilisation in a single unified profiling view. Essential for optimising trillion-parameter model training on the UltraServer.
